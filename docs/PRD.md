## 1. 프로젝트 개요

### 1.1 프로젝트 명
**쉽약 (SHIPyak)** — Solar Pro3 기반 공공문서 평이화 멀티에이전트 플랫폼

### 1.2 포지셔닝
- **본질**: 행정·법률 용어로 쓰인 공공문서를 시민의 눈높이에 맞춰 자동 변환하는 **범용 문서 평이화 엔진**
- **첫 번째 유즈케이스**: 선거 공약집 평이화 (시의성·사회적 임팩트)
- **확장 유즈케이스**: 보험 약관, 금융 설명서, 의료 동의서, 정부 고시·공고

### 1.3 목표
선거 공약집의 행정·법률 용어를 다양한 인구 집단(페르소나)의 언어 수준에 맞춰 자동 변환하는 멀티에이전트 시스템 MVP를 구현한다. 원문의 의미를 보존하면서 가독성을 높이는 것에 집중하며, 정치적 중립성을 구조적으로 보장한다.

### 1.4 문제 정의

| 문제 | 근거 |
|------|------|
| 공약집이 행정·법률 용어로 가득해 일반 시민이 이해하기 어려움 | OECD PIAAC 한국 성인 문해력: 하위 수준 비율 38% |
| 정보취약층(고령자·저학력·외국인)일수록 공약 접근성이 낮음 | 65세 이상 디지털 정보 격차 지수: 전체 대비 63.5% 수준 (NIA 2024) |
| 같은 공약이라도 사람마다 '어려운 지점'이 다름 | 70대는 영어 약자, 30대는 정책 브랜드명, 외국인은 한자어가 걸림 |
| 기존 공약 비교 서비스는 '요약·시각화'에 그침 | 요약해도 여전히 행정 용어 그대로 — 정보격차 해소 불가 |

### 1.5 주요 사용자군
- **1차 사용자 (B2C)**: 유권자 (정보취약층 포함)
- **2차 사용자 (B2B)**: 선관위·지자체·시민단체·언론사·보험사·금융기관
- **해커톤 시연 대상**: 심사위원 (Upstage 관계자 포함)

### 1.6 성공 지표 (MVP 기준)
- 지역 선택 → 후보 공약 → 페르소나별 평이화 결과까지 **탭 4번 이내**
- 원문 대비 평이화 텍스트의 가독성 점수 30% 이상 향상
- 원문과 평이화 텍스트 간 의미 보존율 95% 이상 (수동 검증)
- 모든 후보 공약에 동일한 톤·품질 유지 (중립성 검증)

### 1.7 핵심 차별화
- "공약 비교는 있다. **공약 번역기는 없다.**"
- Nemotron-Personas-Korea(700만 한국인 합성 페르소나) + Solar Pro3 멀티에이전트의 최초 결합
- **사용자가 PDF를 찾을 필요 없음** — 서비스가 공약 데이터를 이미 보유하고, 동네 선택만 하면 즉시 결과

### 1.8 핵심 UX 원칙
> "74세 농촌 어르신이 손녀가 보내준 카톡 링크를 눌러서, 회원가입 없이, PDF 다운로드 없이, 탭 3~4번 만에 우리 동네 후보 공약을 쉬운 말로 읽을 수 있어야 한다."

- 회원가입 없음
- 앱 설치 없음 (모바일 웹)
- PDF 업로드 불필요 (서비스가 공약 보유)
- 탭 4번 이내에 결과 도달

## 2. 설계 원칙 (Safety by Design)

### 2.1 중립성 보장 원칙

| 원칙 | 구현 방법 |
|------|----------|
| **동일 파이프라인** | 모든 후보 공약을 동일 프롬프트·동일 모델·동일 파라미터(temperature 포함)로 처리 |
| **후보 순서 중립** | 후보 표시 순서는 항상 기호순(선관위 공식 기호) |
| **톤 일관성** | 모든 평이화 텍스트에 동일한 톤 가이드라인 적용. 긍정적/부정적 형용사 사용 금지 |
| **평가·추천 금지** | "이 공약이 좋다/나쁘다", "이 후보를 추천" 류의 판단성 문구 일절 생성 금지 |

### 2.2 의미 보존 원칙

| 원칙 | 구현 방법 |
|------|----------|
| **원문 항상 병기** | 평이화 텍스트 옆에 반드시 원문 표시. 원문 삭제·대체 불가 |
| **의역 최소화** | 어휘 교체 + 문장 분할만 허용. 원문에 없는 해석·추론·보충 설명 금지 |
| **수치·고유명사 보존** | 예산 금액, 기간, 법률명, 사업명 등은 원문 그대로 유지 |
| **할루시네이션 방지** | 원문에 없는 구체적 금액·수치·효과 생성 금지 |

### 2.3 면책 고지 원칙

| 위치 | 고지 문구 |
|------|----------|
| 서비스 전체 | "쉽약은 AI가 공약을 쉬운 말로 풀어쓴 것이며, 정확한 내용은 원문을 확인하세요." |
| 모든 결과 화면 | "이 내용은 AI가 풀어쓴 보조 설명입니다. 원문이 정확한 내용이며, 판단은 유권자 본인의 몫입니다." |
| 공유 시 | 공유 카드에 원문 + 평이화 + 면책 고지 3가지 모두 포함 |

## 3. 사용자 인터뷰 질문 자동 생성

### 3.1 정보취약층 유권자 대상

**공약 이해도 파악**
1. 가장 최근 투표 때, 후보의 공약을 어디서 확인하셨나요?
2. 공약집을 읽어보신 적 있나요? 있다면 끝까지 읽으셨나요?
3. 공약에서 이해가 안 되는 단어나 문장이 있었다면 어떤 것이었나요?
4. "SOC 확충", "정주여건 개선" 같은 표현을 들으면 어떤 느낌이 드시나요?
5. 공약을 쉽게 풀어서 설명해주는 사람이나 서비스가 있었다면 도움이 됐을까요?

**행동 패턴**
6. 투표 전 후보 정보를 얼마나 찾아보시나요? (전혀 안 함 / 간단히 / 꼼꼼히)
7. 주로 어떤 기기로 정보를 확인하시나요? (스마트폰 / PC / 종이 공보물)
8. 공약 PDF를 직접 다운로드해서 읽어본 적 있나요?
9. 가족이나 지인에게 "이 후보 공약이 뭐야?"라고 물어본 적 있나요?

**니즈 검증**
10. 동네만 선택하면 후보 공약이 쉬운 말로 바로 나온다면 사용하시겠어요?
11. 이런 서비스를 쓸 때, 가장 걱정되는 점은 무엇인가요? (정확성? 편향?)

### 3.2 시민단체·선관위 관계자 대상

1. 유권자 정보 격차 해소를 위해 현재 어떤 노력을 하고 계신가요?
2. 공약집의 가독성을 높이기 위한 가이드라인이 있나요?
3. 외국인·장애인·고령자 유권자를 위한 별도 자료를 제작하시나요?
4. AI 기반 평이화 도구가 있다면 업무에 활용할 의향이 있으신가요?
5. AI가 공약을 다시 쓸 때, 가장 우려되는 점은 무엇인가요?

**출력**: 후보별 구조화된 공약 JSON DB

**로직**:
1. 정책공약마당에서 2022년 지방선거 당선인 공약 PDF를 수동 다운로드
2. 각 PDF를 Upstage Document Parse API에 전달 → 텍스트 추출
3. 추출된 텍스트를 Upstage Information Extract API에 전달 → 공약별 구조화
4. 구조화 스키마: { "후보명": str, "지역": str, "선거종류": str, "공약목록": [{ "분야": str, "공약명": str, "목표": str, "예산": str }] }
5. 결과를 JSON/SQLite로 저장
6. 이후 사용자 요청 시 이 DB에서 즉시 조회 → Agent 1·2 실시간 실행 불필요

### 10.2 Agent 3: 이해도 평가 (실시간)

**입력**: 사전 파싱된 공약 항목 + 선택된 페르소나 프로필
**출력**: 문장별 이해도 점수 + 어려운 단어 목록

**로직**:
1. Solar Pro3에 시스템 프롬프트로 페르소나 프로필 주입 (Nemotron 서술형 프로필)
2. 프롬프트: "당신은 [74세, 초등학교 졸업, 충남 논산 농업 종사자]입니다. 다음 문장을 읽고 이해할 수 있는지 판단하세요."
3. 국립국어원 어휘 등급 기준 참조
4. 각 문장에 대해 반환: { "이해도": "쉬움|보통|어려움", "어려운_단어": [...], "이유": "..." }

### 10.3 Agent 4: 평이화 텍스트 생성 (실시간)

**입력**: 원문 + 이해도 평가 결과 + 페르소나 프로필
**출력**: 평이화된 텍스트

**로직**:
1. Solar Pro3에 시스템 프롬프트로 국립국어원 "쉬운 공공언어 쓰기 길잡이" 규칙 주입
2. "어려움" 판정된 문장만 재작성 대상으로 선별
3. 재작성 규칙 (의미 보존 원칙 적용):
   - 영어 약자 → 한글 풀어쓰기 + 괄호 원문 병기
   - 한자어 → 순한글 동의어 치환
   - 긴 문장 → 2~3개 짧은 문장으로 분할
   - **금지**: 원문에 없는 해석·추론·보충 설명, 긍정적/부정적 형용사, 구체적 금액·수치 생성
4. 재작성 후 Agent 3으로 재검증 (최대 2회 루프)
5. 2회 후에도 "어려움"이면 그대로 표시 + "이 부분은 AI도 쉽게 풀기 어려운 내용이에요" 안내

### 10.4 Agent 1·2: Document Parse + Info Extract (보조 플로우 전용)

**실행 조건**: PDF 직접 업로드 시에만 실시간 실행
**로직**: v2와 동일 (Upstage Document Parse → Information Extract → 구조화)

### 10.5 오케스트레이터

**로직**:
1. 메인 플로우: DB 조회 → 페르소나 선택 대기 → Agent 3 → Agent 4 → 결과
2. 보조 플로우: Agent 1 → Agent 2 → 페르소나 선택 대기 → Agent 3 → Agent 4 → 결과
3. 중립성 검증: 동일 선거구 내 모든 후보에 동일 temperature·max_tokens·시스템 프롬프트 적용 로그 기록
4. 에러 시 해당 Agent만 재시도 (최대 2회)
5. 진행 상황 프론트엔드에 스트리밍

## 12. 발표 방어 로직

> 발표 중간에 팀이 먼저 언급해야 할 5가지.

**1번** (중립성): "모든 후보 공약을 동일 프롬프트, 동일 모델, 동일 파라미터로 처리합니다."

**2번** (원문 존중): "원문은 항상 함께 보여드립니다. AI는 보조 설명이고, 판단은 유권자 본인의 몫입니다."

**3번** (범용성): "선거는 첫 번째 유즈케이스일 뿐입니다. 진짜 시장은 보험 약관, 금융 설명서, 의료 동의서입니다."

**4번** (한계 인정): "페르소나 시뮬레이션은 완벽한 대리가 아닙니다. 다양한 관점 검토의 시작점이고, 실사용자 테스트로 보정할 것입니다."

**5번** (윤리): "AI가 유권자 판단에 개입해서는 안 됩니다. 의역 최소화, 원문 병기, 톤 중립화를 설계 원칙으로 삼았습니다."

## 15. MVP 이후 추가 개선사항
