# 면접 에이전트 데모 실행 결과

- **실행일**: 2026-05-23
- **대상 파이프라인**: 면접 LangGraph (`resume → trend → stress → judge`)
- **실행 방법**: `cd backend && uv run python scripts/demo_interview.py`
- **LLM**: Upstage Solar (`solar-pro3`)
- **비고**: Redis·HTTP 없이 그래프를 직접 구동했으며, 지원자 답변은 샘플로 시뮬레이션함. 따라서 아래 점수는 시스템 동작 검증용이며 실제 지원자 평가가 아님.

| 항목 | 값 |
|------|-----|
| 회사 | 테크스타트업 |
| 직무 | 백엔드 개발자 |
| 라운드 수 | 3 (데모용 — 실서비스 기본값 8) |

## 면접 진행

**R0 · resume 면접관**
질문: 모놀리식에서 MSA로 전환하는 과정에서 구체적으로 어떤 서비스를 분리해 냈으며, 각 서비스 간 데이터 일관성 유지를 위해 어떤 방식을 도입했는지 설명해 주세요.
답변(샘플): MSA 전환 프로젝트를 주도해 주문 서비스의 응답 지연을 40% 줄였습니다. 부하 테스트로 병목을 찾아 캐시 계층과 비동기 처리를 도입했습니다.

**R1 · trend 면접관**
질문: 백엔드 개발 분야에서 최근 주목받는 서버리스 아키텍처의 확산 속에서, 테크스타트업과 같은 빠른 개발 주기가 요구되는 환경에서 서비스 설계 시 고려해야 할 기술적 장점과 잠재적 한계는 무엇이라고 생각하십니까?
답변(샘플): 서비스 간 데이터 일관성은 이벤트 기반 메시징과 보상 트랜잭션으로 처리했습니다.

**R2 · stress 면접관**
질문: 이벤트 기반 메시징과 보상 트랜잭션 적용 시 발생할 수 있는 데이터 최종 일관성 지연 문제를 어떻게 모니터링하고 대응했는지 구체적으로 설명해 주시겠어요?
답변(샘플): 최근에는 LLM 기반 기능이 백엔드 설계에 영향을 준다고 보고, 비동기 처리와 비용 모니터링을 함께 고민하고 있습니다.

## 종합 피드백

> 지원자는 MSA 전환 경험과 기술적 문제 해결 능력을 보여주었으나, 질문에 대한 답변의 구체성과 깊이가 다소 부족합니다. 예를 들어, '어떤 서비스를 분리했는지'나 '보상 트랜잭션의 구체적인 구현 방식'에 대한 설명이 미흡했습니다. 또한 LLM 도입 시 고려해야 할 비용/성능 트레이드오프에 대한 통찰이 필요하지만, 현재 답변에서는 트렌드에 대한 표면적 이해만 드러납니다. 경험을 더 체계적으로 정리하고, 기술적 선택 시 trade-off 분석을 명확히 한다면 평가 점수가 크게 향상될 것입니다.
